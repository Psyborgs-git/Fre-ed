## The knowledge cut-off problem

Every language model is frozen at the moment it finished training. Ask GPT-4 about events
from last month, a niche internal document, or your company's codebase and it will either
hallucinate or admit it doesn't know.

**Retrieval-Augmented Generation (RAG)** is the practical solution: give the model access
to an external knowledge base at *inference time*. No retraining required.

The 3D scene shows the full pipeline: a query sphere fires similarity rays into a corpus
of floating document rectangles, the top-3 most similar documents glow amber and drift
into a context column, then a violet LLM box generates tokens conditioned on both the
query and the retrieved context.

<Callout type="note">
Scroll to watch each stage: similarity search → top-k retrieval → context assembly →
generation. The key insight is that the LLM's weights never change — only its *input*
grows to include the retrieved documents.
</Callout>

## The RAG pipeline

```
Query → Embed query → Similarity search over vector DB → Retrieve top-k → Augment prompt → Generate
```

Three components work together:

1. **Encoder** — maps both queries and documents into the same embedding space so
   similarity can be computed as a vector distance.
2. **Vector database** — stores pre-computed document embeddings; supports fast approximate
   nearest-neighbour search (ANN).
3. **LLM** — generates the final answer, conditioned on the retrieved chunks injected into
   its prompt.

## Embedding and similarity search

Documents are chunked (e.g., 512 tokens each) and encoded into dense vectors by an
embedding model (e.g., `text-embedding-3-small`, `bge-m3`, `e5-large`).

At query time, the same encoder maps the query to a vector $\mathbf{q}$. Relevance to
each document chunk $\mathbf{d}_i$ is measured with **cosine similarity**:

$$
\text{sim}(\mathbf{q}, \mathbf{d}_i)
= \frac{\mathbf{q} \cdot \mathbf{d}_i}{\|\mathbf{q}\|\,\|\mathbf{d}_i\|}
$$

The top-k chunks with the highest similarity scores are retrieved and injected into the
prompt.

<CodeBlock language="python">
{`from sentence_transformers import SentenceTransformer
import numpy as np

encoder = SentenceTransformer("BAAI/bge-small-en-v1.5")

# Offline: encode and store document chunks
docs = ["Paris is the capital of France.",
        "Python was created by Guido van Rossum.",
        "The transformer architecture was introduced in 2017."]
doc_embeddings = encoder.encode(docs, normalize_embeddings=True)

# Online: encode the query and retrieve top-k
query = "Who invented Python?"
q_emb = encoder.encode([query], normalize_embeddings=True)

scores = (q_emb @ doc_embeddings.T)[0]   # cosine sim (normalized → dot product)
top_k = np.argsort(scores)[::-1][:2]     # indices of top-2 matches
retrieved = [docs[i] for i in top_k]
print(retrieved)  # ['Python was created by Guido van Rossum.', ...]`}
</CodeBlock>

## Vector databases in practice

A flat NumPy search is fine for thousands of documents. At millions of chunks you need an
**approximate nearest-neighbour (ANN) index** — typically HNSW or IVF:

| Database | Index type | Hosted | Notes |
|---|---|---|---|
| FAISS | IVF, HNSW | Self | Meta's library; in-process |
| Pinecone | Proprietary | Cloud | Managed, serverless tier |
| Weaviate | HNSW | Both | Multi-modal, GraphQL API |
| Qdrant | HNSW | Both | Rust core, fast filtering |
| pgvector | HNSW, IVF | Self (Postgres) | SQL-native; good for small-medium scale |
| ChromaDB | HNSW | Self | Python-native; popular for prototyping |

<CodeBlock language="python">
{`# FAISS example — 1M embeddings at 768 dims
import faiss

d = 768          # embedding dimension
index = faiss.IndexHNSWFlat(d, 32)   # HNSW with M=32 neighbours
index.add(doc_embeddings.astype("float32"))

# Search
k = 5
D, I = index.search(q_emb.astype("float32"), k)   # distances, indices
retrieved_chunks = [docs[i] for i in I[0]]`}
</CodeBlock>

## Chunking and context window limits

A document rarely fits in the LLM's context window. **Chunking strategy** matters:

- **Fixed-size chunks** — simple; 256–512 tokens, with 10–20 % overlap to avoid cutting
  mid-sentence.
- **Semantic chunking** — split on sentence or paragraph boundaries; more coherent but
  harder to implement.
- **Late chunking** — encode the full document, then split embeddings; preserves
  long-range context in the embedding (recent approach).

<Callout type="warning">
Naive chunking breaks tables, code blocks, and lists. Always test retrieval quality on a
representative sample of your document types — especially for structured content.
</Callout>

After retrieval, chunks are inserted into the prompt:

<CodeBlock language="python">
{`SYSTEM = "Answer using only the provided context."

def build_prompt(query, retrieved_chunks):
    context = "\n\n---\n\n".join(retrieved_chunks)
    return f"""{SYSTEM}

Context:
{context}

Question: {query}
Answer:"""

# The LLM never sees documents it didn't retrieve
prompt = build_prompt("Who invented Python?", retrieved_chunks)`}
</CodeBlock>

## RAG vs fine-tuning — when to use which

| Dimension | RAG | Fine-Tuning |
|---|---|---|
| Knowledge updates | Instant (update DB) | Requires retraining |
| Private/external docs | Yes | Baked into weights |
| Hallucination risk | Lower (grounded in retrieved text) | Higher (model can confabulate) |
| Cost | Inference + retrieval | Training compute |
| Reasoning style | Primarily extractive | Can learn new styles/tasks |
| Citation / source | Easy (log retrieved chunks) | Hard |

<Callout type="tip">
In practice, RAG and fine-tuning are complementary, not competing. Fine-tune the LLM for
*format and style* (e.g., output JSON, match company voice), then add RAG for
*knowledge* (current facts, proprietary docs). Using both together is now the industry
default for production LLM applications.
</Callout>

## Advanced RAG patterns

Basic RAG has failure modes: the retrieved chunks may not contain the answer, or the
answer requires reasoning across multiple chunks. Several patterns address this:

- **HyDE (Hypothetical Document Embeddings)** — generate a hypothetical answer first, then
  embed *that* and use it as the retrieval query. Closes the semantic gap between short
  queries and long documents.
- **CRAG (Corrective RAG)** — grade retrieved documents for relevance; if all are poor,
  fall back to web search. Reduces noise from irrelevant retrievals.
- **Self-RAG** — the LLM decides *when* to retrieve and grades its own output. Interleaves
  generation and retrieval rather than treating retrieval as a preprocessing step.
- **Multi-step RAG** — decompose complex questions into sub-questions, retrieve for each,
  then synthesise the final answer.
